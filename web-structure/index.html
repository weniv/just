딱 필요한 만큼 — 바이브 코딩을 위한 웹 서비스 구조

웹 구조 도식 모음

AI에게 “이 구조로 만들어줘”라고 말할 수 있게 되는 것이 목표인 책입니다. 본문 곳곳에 흩어져 있는 13장의 머메이드 도식만 모아 한 페이지에서 훑어볼 수 있게 정리했습니다.

1. 왜 웹 구조부터인가

1.1

AI에게 던지면 따라오는 질문

AI에게 “할 일 관리 앱 만들어줘”라고 던지면 곧장 “무엇이 필요한가요?”라는 질문이 돌아옵니다. 이 책은 그 질문에 스스로 답할 수 있는 구조 감각을 키우는 것이 목표입니다.

flowchart LR A["내가 원하는 서비스"] --> B["AI"] B --> C{"내가 만들 서비스
무엇이 필요한가?"} C --> D["프론트엔드"] C --> E["백엔드"] C --> F["데이터베이스"] C --> G["인프라"] C --> H["AI 모델 API / 지도 API 등
(외부 백엔드)"]

2. 한 장의 지도 — 5가지 구조

2.1

모든 구성요소를 한 장에

대부분의 웹 서비스를 간소화한 한 장의 그림. 책 전반에서 계속 등장하며, 만들고 싶은 서비스가 어디까지 필요한지 선을 그어볼 수 있는 ‘지도’가 됩니다.

2.2

프론트엔드만 개발

HTML·CSS·JS 파일만 있으면 동작하는 가장 단순한 구조. 회사 소개 페이지, 계산기, 틱택토 같은 단순 도구가 여기에 들어갑니다.

flowchart LR USER["사용자
(브라우저)"] -- "1 - 주소 접속" --> WEB["Web Server
(프론트엔드 소스코드
HTML / CSS / JS)"] WEB -- "2 - 파일 전송" --> USER
2.3

모놀리식 (DB는 따로)

Django 같은 풀스택 프레임워크. 한 프로젝트에 화면과 처리 로직이 같이 들어 있고, DB는 따로 두는 구조입니다. 페이지 요청이 오면 백엔드가 DB를 조회해 완성된 HTML을 만들어 보냅니다.

flowchart LR USER["사용자"] -- "1 - 페이지 요청" --> SERVER["Django 서버
(Web Server + App Server 한 몸)"] SERVER -- "2 - 데이터 조회" --> DB[("DB Server
(데이터)")] DB -- "3 - 결과" --> SERVER SERVER -- "4 - 완성된 HTML" --> USER
2.4

모놀리식 (한 몸)

Web/App/DB가 한 몸인 구조. 1인 개발이나 작은 팀, 일 이용자 1만명 정도까지의 중규모 서비스에 잘 어울립니다. 유지보수가 가장 편한 패턴이기도 합니다.

flowchart LR USER["사용자
(브라우저)"] -- "1 - 페이지 요청" --> SERVER["Django 서버
(Web Server +
App Server +
DB Server 한 몸)"] SERVER -- "2 - 완성된 HTML" --> USER
2.5

마이크로서비스 — 프론트/백 분리

화면 따로, 처리 로직 따로 만드는 구조. ‘프론트엔드 팀’과 ‘백엔드 팀’이 나뉜 큰 조직에서 흔히 선택합니다. 두 프로젝트는 API라는 약속된 형식으로만 데이터를 주고받습니다.

flowchart LR USER["사용자
(브라우저)"] -- "1 - 화면 요청" --> FE["Web Server
(프론트엔드 소스코드)"] FE -- "2 - 프론트엔드 코드 전송" --> USER USER -- "3 - API 호출
(브라우저에서 실행되는
프론트엔드 코드가)" --> BE["App Server
(백엔드 소스코드)"] BE -- "4 - 데이터 조회" --> DB[("DB Server
(데이터)")] DB -- "5 - 결과" --> BE BE -- "6 - JSON 응답
(화면 일부만 갱신)" --> USER
2.6

프론트엔드 + 외부 API

백엔드 자리를 다른 회사 API로 대체. 지도 API, ChatGPT API, 공공데이터 포털, 날씨 위젯처럼 외부 데이터·기능을 빌려 쓰는 화면에 잘 맞습니다.

flowchart LR USER["사용자
(브라우저)"] -- "1 - 화면 요청" --> FE["Web Server
(프론트엔드 소스코드)"] FE -- "2 - 프론트엔드 코드 전송" --> USER USER -- "3 - 외부 API 호출
(브라우저에서 실행되는
프론트엔드 코드가)" --> EXT["외부 App Server
(OpenAI / 지도 / 공공데이터 등)"] EXT -- "4 - 응답
(화면 갱신)" --> USER
2.7

BaaS — 백엔드를 통째로 빌리기

회원가입·로그인·DB·파일 저장소를 한 묶음으로 빌려주는 서비스(Firebase, Supabase 등). 백엔드 코드를 거의 짜지 않고도 회원제 서비스를 1~2주 안에 띄울 수 있습니다.

flowchart LR USER["사용자
(브라우저)"] -- "1 - 화면 요청" --> FE["Web Server
(프론트엔드 소스코드)"] FE -- "2 - 프론트엔드 코드 전송" --> USER USER -- "3 - 회원/데이터 요청
(브라우저에서 실행되는
프론트엔드 코드가)" --> BAAS["BaaS
(App Server + DB Server 한 묶음)"] BAAS -- "4 - 결과
(화면 갱신)" --> USER

3. 구성요소 들여다보기

3.1

요청과 응답

브라우저가 서버에 보내는 행동을 ‘요청(Request)’, 서버가 돌려주는 묶음을 ‘응답(Response)’이라 부릅니다. 브라우저는 받아온 HTML/CSS/JS를 해석해 화면을 그립니다.

flowchart LR subgraph SERVER["서버 (Server)"] BE["백엔드 언어
(Python, Java,
Node.js, PHP 등)"] DB[("데이터베이스")] BE <--> DB end subgraph BROWSER["브라우저 (Browser)"] HTML["③ 전송 받은 HTML"] CSS["③ 전송 받은 CSS"] JS["③ 전송 받은 JavaScript"] PAGE["④ 완성된 화면"] HTML --> PAGE CSS --> PAGE JS --> PAGE end BROWSER -->|"① 요청 (Request)"| SERVER SERVER -->|"② 응답: HTML, CSS, JS 전달"| BROWSER
3.2

HTML / CSS / JavaScript의 역할

HTML은 뼈대, CSS는 꾸밈, JavaScript는 움직임. 이 셋이 합쳐져 우리가 보는 한 장의 화면이 만들어집니다.

flowchart LR HTML["HTML
(뼈대/내용)"] --> PAGE["완성된 화면"] CSS["CSS
(꾸밈/배치)"] --> PAGE JS["JavaScript
(움직임/반응)"] --> PAGE
3.3

도메인과 DNS

사람이 외운 google.com을 IP 주소로 바꿔주는 게 DNS. 우리는 이름으로 접속하지만 실제로는 그 이름이 가리키는 숫자 주소로 컴퓨터를 찾아갑니다.

flowchart LR USER["사용자가
google.com 입력"] --> DNS["전화번호부
(DNS)"] DNS -- "google.com은
142.250.196.110 입니다" --> USER USER -- "142.250.196.110으로 접속" --> SERVER["구글 서버"]
3.4

FastAPI 서버 흐름

주소(/)와 (/about)에 따라 다른 HTML이 돌아오는 모습. 백엔드는 사용자의 요청 주소에 맞춰 응답을 만들어 보내는 일을 합니다.

sequenceDiagram participant BROWSER as 브라우저 participant SERVER as FastAPI 서버
(002.py) BROWSER->>SERVER: 1. GET / SERVER->>BROWSER: 2. HTML 응답 BROWSER->>SERVER: 3. GET /about SERVER->>BROWSER: 4. 다른 HTML 응답
3.5

데이터베이스가 필요한 이유

서버를 껐다 켜도 데이터가 남아 있어야 한다면 DB가 필요합니다. 어제 작성한 글을 오늘 다시 읽을 수 있는 이유가 여기 있습니다.

sequenceDiagram participant USER as 사용자 participant BE as 백엔드
(Python) participant DB as 데이터베이스 participant USER2 as 다른 사용자
(다음 날 접속) USER->>BE: 1. 글 작성 요청 BE->>DB: 2. 저장 DB->>BE: 3. 저장 완료 BE->>USER: 4. 응답 USER2->>BE: 5. 글 목록 요청 BE->>DB: 6. 조회 DB->>BE: 7. 어제 작성된 글 BE->>USER2: 8. 응답
